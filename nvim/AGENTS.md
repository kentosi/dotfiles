# AGENTS.md

This file provides guidance to AI coding agents when working with code in this repository.

## What this is

A Neovim configuration written entirely in Lua, using `lazy.nvim` as the plugin manager. There is no build step, no test suite, and no linter configured — changes are validated by launching Neovim with this config and exercising the affected behavior.

## Running / testing changes

This config is not (yet) the user's active `~/.config/nvim` — it lives at a separate path and must be loaded explicitly:

```sh
NVIM_APPNAME=<this-dir-name> nvim              # interactive, loads this config via XDG config lookup
NVIM_APPNAME=<this-dir-name> nvim --headless -c 'qa'   # headless smoke test — confirms no load errors, triggers lazy.nvim plugin install/sync
```

Useful in-editor checks after a change:
- `:Lazy` — confirm all plugin specs loaded with no errors, check install/update status
- `:checkhealth` — dependency/health checks for plugins like telescope and treesitter
- `:messages` — check for runtime errors after startup

When adding or editing a plugin spec, do a headless run first (`nvim --headless -c 'qa'`) to catch load-time Lua errors before testing interactively — errors during plugin `config` functions print to stdout/stderr and are easy to miss otherwise.

## Architecture

Entry point is `init.lua`, which just requires three modules in order:

```lua
require("config.options")   -- vim.opt settings (non-plugin editor behavior)
require("config.keymaps")   -- vim.g.mapleader + non-plugin keymaps
require("config.lazy")      -- bootstraps lazy.nvim, then loads lua/plugins/*
```

This order matters: `mapleader` must be set (in `keymaps.lua`) *before* `config.lazy` runs, since plugin `keys` specs (e.g. `<leader>ff`) are resolved against `mapleader` at plugin-load time.

**`lua/config/`** — non-plugin editor setup plus the plugin manager bootstrap:
- `options.lua` / `keymaps.lua` — plain `vim.opt`/`vim.keymap.set` calls, no plugin dependencies
- `lazy.lua` — standard lazy.nvim self-cloning bootstrap, then `require("lazy").setup({ spec = { { import = "plugins" } } })`, which auto-loads every spec file under `lua/plugins/`

**`lua/plugins/`** — one file per plugin (or tightly related pair), each returning a lazy.nvim plugin spec table. This is the pattern to follow when adding a new plugin: create `lua/plugins/<name>.lua` returning `{ "author/repo", ... }`; no other file needs to change, since `lazy.lua` imports the whole directory.

**`lua/util/`** — standalone helper modules that plugin specs `require()` from their `config` function, for logic substantial enough to not belong inline in a plugin spec (e.g. `vimwiki_gitpush.lua`, which registers the `:VimWikiGitPush` user command used from `plugins/vimwiki.lua`).

**`lazy-lock.json`** — committed lockfile pinning exact plugin commits; regenerated by lazy.nvim (`:Lazy sync`/`:Lazy update`), not hand-edited.

### Notable non-obvious choices

- `nvim-treesitter` is pinned to `branch = "master"` in `plugins/treesitter.lua`. The `main` branch is a breaking rewrite that removed the `nvim-treesitter.configs` module this config's `config` function depends on — do not remove the branch pin without rewriting that file's setup call for the new API.
- Several plugins intentionally replace older Vimscript equivalents from a prior config rather than porting them 1:1: `gitsigns.nvim` (not vim-gitgutter), `lualine.nvim` (not lightline.vim + vim-gitbranch), and Neovim's built-in `gc`/`gcc` commenting (not vim-commentary/tpope-commentary). Prefer maintained Lua-native plugins over their Vimscript predecessors when adding similar functionality.
